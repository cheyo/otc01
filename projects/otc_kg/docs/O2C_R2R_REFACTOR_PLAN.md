# O2C + R2R 标准方案重构说明

本次重构目标不是最小可行版本，而是将当前 OTC 原型升级为更标准的 O2C（Order to Cash）+ R2R（Record to Report）业务模型。

## 重构原则

1. 保留 Oracle ETRM 风格命名
2. 覆盖主数据、订单、履约、开票、应收、收款、核销、总账、子账、余额、风险增强
3. 优先做标准分层，后续再逐步迁移造数脚本、导入脚本、Neo4j 映射与 API
4. 当前阶段先交付标准版 DDL 蓝图与重构计划，再逐步切换运行时数据管道

## 本次新增的核心文件

- `mysql/schema_v2_o2c_r2r.sql`
  - 标准版 O2C + R2R DDL 蓝图
- `docs/O2C_R2R_REFACTOR_PLAN.md`
  - 重构范围、目标、分阶段计划

## 方案覆盖范围

### O2C
- 客户主数据
- 客户站点与用途
- 银行账户与信用档案
- 订单头/行
- 订单冻结、价格调整
- 发运单头、发运明细、发运分配
- 发票头/行、税行
- 应收计划、调整、贷项
- 收款、核销、银行对账

### R2R
- 账簿
- 会计期间
- 科目组合
- 子账会计头/行/分配链接
- 总账凭证头/行
- 导入引用桥接
- 科目余额

### 风险增强层
- 审批日志
- 字段变更日志
- 风险事件

## 建议实施顺序

### Phase 1: Schema 标准化
- 完成 `schema_v2_o2c_r2r.sql`
- 明确旧表 -> 新标准表映射

### Phase 2: Generator 升级
- 升级主数据与订单造数
- 新增发运单头、站点、税行、总账、子账样本

### Phase 3: Load / Export 升级
- 升级 MySQL 导入 SQL
- 升级 Neo4j 导出映射

### Phase 4: API / Frontend 升级
- 新增 O2C-R2R 联查
- 新增凭证与余额可视化
- 新增从订单到总账的链路查询

## 当前注意事项

当前运行中的原型仍以 `schema_v1.sql` 和现有样本管道为主；`schema_v2_o2c_r2r.sql` 是标准方案重构起点，接下来需要逐步切换各模块。
