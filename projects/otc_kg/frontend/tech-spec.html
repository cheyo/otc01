OTC 风险知识图谱应用技术方案详解

本页汇总当前 OTC 应用的整体架构、数据模型、采样规模、Neo4j 图谱设计、查询能力与部署方式。

1. 应用目标

本应用基于 Oracle ETRM 风格的 OTC（Order to Cash）流程，构建一套从 MySQL 标准表到 Neo4j 知识图谱，再到 Web 风险分析与自然语言查询的完整原型系统，用于识别潜在财务风险，例如共享银行账户、逾期应收、高风险客户订单、未开票发货、大额折扣、超授信订单等。

2. 总体技术架构

  • 关系数据库： MySQL 8.0，承载 Oracle ETRM 风格 OTC 业务表
  • 图数据库： Neo4j 5.26，承载业务图谱与风险关系
  • 后端： FastAPI，提供风险查询 API、自然语言查询 API
  • 前端： HTML + Vue 3 + Axios，提供查询界面与结果展示
  • 数据生成： Python 脚本，生成中国化客户、订单、发票、收款样本

4. 数据采样与生成策略

  • 当前为 原型样本集，不是最终 50 万级正式版
  • 客户、组织名、地址、银行名采用 中国命名习惯
  • 业务链按 客户 → 订单 → 发运 → 发票 → 应收 → 收款 → 核销 自动派生
  • 风险样本采用模板化注入，如共享银行账户、大额折扣、未开票发货、高逾期应收等
  • 主链路校验结果：孤儿订单行 0、孤儿发票行 0、断链核销 0

已注入的典型风险特征

  • 共享银行账户：7 组客户关系
  • 大额折扣：折扣超过 20% 的订单行
  • 逾期应收：存在超过 90 天甚至 700 天以上的样本
  • 未开票发货：部分 DeliveryDetail 没有对应 InvoiceLine
  • 审批链：部分高金额订单具备审批记录

6. 当前已实现的风险查询

  • 共享银行账户客户群
  • 高逾期应收客户列表
  • 高风险客户订单
  • 大额折扣订单
  • 未开票发货
  • 超信用额度订单
  • 审批链路穿透
  • 客户 OTC 全链路追踪

7. 自然语言查询实现方式

  • 当前版本采用 规则/关键词意图识别，而不是正式 LLM 直出 Cypher
  • 输入自然语言后，先识别意图，再映射到预定义风险查询
  • 支持返回自然语言解释、结果摘要、表格结果
  • 后续可升级为真正的 LLM → 结构化参数 / Cypher 模式

8. 当前访问方式

  • 前端：http://服务器IP:8080
  • 后端健康检查：http://服务器IP:8000/health
  • 后端统计：http://服务器IP:8000/stats

9. 后续可扩展方向

  • 扩展到 50 万级以上正式数据
  • 增加退货、贷项、坏账、调整等更多财务对象
  • 引入真实 LLM 做自然语言 → 查询转换
  • 补充图谱可视化、路径探索、风险评分卡
  • 统一前后端单端口部署，避免跨端口访问问题
